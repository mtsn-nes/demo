Подключение к устройствам по SSH и CLI
Как работать с устройствами через терминал Proxmox и SSH

Доступ через консоль Proxmox

На первом этапе (до настройки SSH) необходимо работать через веб-консоль Proxmox:

  1. Откройте браузер и перейдите на веб-интерфейс Proxmox: https://<IP_Proxmox>:8006
  2. Выберите нужную виртуальную машину в левом дереве (например, ISP)
  3. Нажмите кнопку Console → откроется терминальное окно
  4. Войдите под root: логин root, пароль 123123

Подключение по SSH к Linux-серверам (ISP, HQ-SRV, BR-SRV)

После настройки сетевых интерфейсов можно подключаться по SSH с вашего рабочего ПК или с другой ВМ:

# Стандартное подключение (порт 22, до выполнения задания 5)
ssh root@192.168.100.2          # → HQ-SRV
ssh root@192.168.255.2          # → BR-SRV
ssh root@172.16.1.1             # → ISP (если доступен)

# После выполнения задания 5 (порт изменён на 2026, только sshuser)
ssh -p 2026 sshuser@192.168.100.2   # → HQ-SRV
ssh -p 2026 sshuser@192.168.255.2   # → BR-SRV

📌 Как определить имя интерфейса На Linux (ISP, HQ-SRV, BR-SRV, HQ-CLI):
Выполните команду ip a или nmcli con show.
Интерфейсы обычно называются ens18, ens19, ens20 и т.д. — в зависимости от порядка подключения виртуального NIC.
В выводе nmcli con show вы увидите NAME и DEVICE — используйте значение из столбца NAME в командах nmcli con mod.

На EcoRouter (HQ-RTR, BR-RTR):
Выполните sh int stat — вы увидите список интерфейсов вида gi1/0/1, gi1/0/2 и т.д.
Определить, какой интерфейс куда смотрит, можно по MAC-адресу: сравните MAC в выводе sh int stat с MAC, указанным в настройках сетевой карты Proxmox (вкладка Hardware → Network Device).

Доступ в интернет через ISP
Настройка WAN-интерфейса ISP и NAT (iptables)
Задание Настроить адресацию на ISP. Интерфейс к провайдеру — DHCP. Интерфейсы к HQ-RTR (172.16.1.0/28) и BR-RTR (172.16.2.0/28). Настроить NAT (MASQUERADE) для обоих маршрутизаторов.

2.1 Настройка WAN-интерфейса ISP (DHCP)

# Смотрим список интерфейсов
nmcli con show
ip a

# Интерфейс, смотрящий наружу (например ens18) переводим на DHCP
nmcli con mod ens18 ipv4.method auto
nmcli con up ens18

# Проверяем, что получили адрес
ip a show ens18
# Ожидаем: inet X.X.X.X/... — адрес от провайдера Proxmox

2.2 Настройка интерфейса ISP в сторону HQ-RTR

# Интерфейс ens19 (уточните по ip a — второй NIC)
nmcli con mod ens19 \
  ipv4.method manual \
  ipv4.addresses 172.16.1.1/28

nmcli con up ens19

# Проверка
ip a show ens19
# Ожидаем: inet 172.16.1.1/28

2.3 Настройка интерфейса ISP в сторону BR-RTR

# Интерфейс ens20 (уточните по ip a — третий NIC)
nmcli con mod ens20 \
  ipv4.method manual \
  ipv4.addresses 172.16.2.1/28 \
  ipv4.dns "77.88.8.8"

nmcli con up ens20

2.4 Включение IP Forwarding

Чтобы ISP мог пересылать пакеты между интерфейсами, нужно включить IP-форвардинг:

# Открываем файл конфигурации ядра
nano /etc/sysctl.conf

Добавьте в конец файла строку:

net.ipv4.ip_forward = 1

Сохраните (Ctrl+O → Enter → Ctrl+X). Применяем без перезагрузки:

sysctl -p
# Ожидаем вывод: net.ipv4.ip_forward = 1

2.5 Настройка NAT (MASQUERADE) на ISP

# Устанавливаем iptables-services
dnf install iptables-services -y

# Разрешаем форвардинг трафика (удаляем запрещающее правило)
iptables -D FORWARD 1

# Добавляем MASQUERADE для сети HQ-RTR → интернет
# ens18 — WAN-интерфейс ISP (смотрящий в интернет)
iptables -t nat -A POSTROUTING -s 172.16.1.0/28 -o ens18 -j MASQUERADE

# Добавляем MASQUERADE для сети BR-RTR → интернет
iptables -t nat -A POSTROUTING -s 172.16.2.0/28 -o ens18 -j MASQUERADE

# Сохраняем правила (чтобы восстанавливались после перезагрузки)
iptables-save > /etc/sysconfig/iptables

# Включаем и запускаем iptables-services
systemctl enable --now iptables

# Проверяем правила NAT
iptables -t nat -L POSTROUTING -n -v
📌 Имя WAN-интерфейса ISP Вместо ens18 подставьте фактическое имя интерфейса, который получил адрес от провайдера (смотрите по ip a — интерфейс с DHCP-адресом).
✅ Проверка связи HQ-RTR → интернет После выполнения задания 2 и настройки IP на HQ-RTR (задание 1), выполните с HQ-RTR:
ping 8.8.8.8 — должен пройти через ISP в интернет.

Локальные учётные записи
Пользователи sshuser и net_admin
Задание Создать пользователя sshuser (пароль P@ssw0rd, UID 2026, sudo без пароля) на HQ-SRV и BR-SRV. Создать пользователя net_admin (пароль P@ssw0rd, максимальные привилегии) на HQ-RTR и BR-RTR.

3.1 Создание пользователя sshuser на HQ-SRV и BR-SRV

Выполнить на обоих серверах (HQ-SRV и BR-SRV):

# Создаём пользователя с UID 2026, домашней директорией и bash-оболочкой
useradd -m -s /bin/bash -u 2026 -U sshuser

# Устанавливаем пароль
passwd sshuser
# Введите: P@ssw0rd (дважды)

Настройка sudo без пароля — открываем файл sudoers через visudo (безопасный редактор):

visudo

В открывшемся редакторе найдите строку root ALL=(ALL) ALL и после неё добавьте:

sshuser ALL=(ALL) NOPASSWD: ALL

Сохраните и выйдите (Ctrl+O → Enter → Ctrl+X, или :wq если vi).

# Проверяем
id sshuser
# Ожидаем: uid=2026(sshuser) gid=2026(sshuser) groups=2026(sshuser)

# Проверяем sudo
su - sshuser
sudo whoami
# Ожидаем: root (без запроса пароля)

3.2 Создание пользователя net_admin на HQ-RTR и BR-RTR (EcoRouter)

Выполнить на обоих маршрутизаторах:

# Создаём пользователя с максимальными привилегиями (privilege 15)
username net_admin
password P@ssw0rd
privilege 15
end
commit
confirm
✅ Ожидаемый результат Можно войти в EcoRouter командой ssh net_admin@172.16.1.2 с паролем P@ssw0rd и получить полный доступ.

Безопасный удалённый доступ SSH
Порт 2026, только sshuser, 2 попытки входа, баннер
Задание Настроить SSH на HQ-SRV и BR-SRV: порт 2026, разрешить только sshuser, максимум 2 попытки входа, баннер «Authorized access only».

Выполнить на обоих серверах (HQ-SRV и BR-SRV):

5.1 Создание файла с баннером

# Создаём файл баннера
nano /etc/banner

Введите текст баннера:

Authorized access only

Сохраните (Ctrl+O → Enter → Ctrl+X).

5.2 Редактирование конфигурации SSH

# Открываем конфиг SSH
nano /etc/ssh/sshd_config

Найдите и измените (или добавьте) следующие параметры:

# Изменить порт с 22 на 2026
Port 2026

# Разрешить вход только пользователю sshuser
AllowUsers sshuser

# Максимум 2 попытки ввода пароля
MaxAuthTries 2

# Включить баннер
Banner /etc/banner

Сохраните файл.

5.3 Открытие порта в firewalld и перезапуск SSH

# Добавляем порт 2026 в firewall
firewall-cmd --permanent --add-port=2026/tcp
firewall-cmd --reload

# Перезапускаем SSH
systemctl restart sshd

# Проверяем, что SSH слушает порт 2026
ss -tlnp | grep 2026
# Ожидаем: LISTEN 0 ... 0.0.0.0:2026
✅ Проверка подключения С другой машины: ssh -p 2026 sshuser@192.168.100.2
Должен появиться баннер «Authorized access only», затем запрос пароля.

# Создаём GRE-туннель с номером 10

# IP-адрес туннельного интерфейса

# Включаем туннель
enable
exit

# Применяем
end
commit
confirm

6.2 Настройка туннеля на BR-RTR

tunnel GRE 10
ip firewall disable
local address 172.16.2.2
remote address 172.16.1.2
ip address 10.10.10.2/28
enable
exit
end
commit
confirm

⚠
Возможные ошибки и их решение
Диагностика и устранение типичных проблем
❌ Нет пинга между устройствами в одной сети
Причина: неверно указана маска, неправильное имя интерфейса, интерфейс не поднят.
# Проверьте состояние интерфейса
ip a
nmcli dev status

# Поднимите интерфейс вручную
nmcli con up ens18

# Проверьте таблицу маршрутизации
ip route show
❌ HQ-CLI не получает адрес по DHCP
Причина: DHCP-сервер не запущен на HQ-RTR, или пул настроен неверно, или HQ-CLI остался со статическим IP.
# На HQ-RTR — проверяем пул
sh ip dhcp-server pool
sh ip dhcp-server binding

# На HQ-CLI — принудительно обновляем аренду
nmcli con down ens18
nmcli con up ens18

# Смотрим, какой адрес получен
ip a show ens18
❌ Нет интернета с HQ-SRV / HQ-CLI
Причина: не настроен NAT на ISP или на HQ-RTR, не включён ip_forward на ISP, неверный маршрут по умолчанию.
# На ISP — проверяем форвардинг
sysctl net.ipv4.ip_forward
# Должно быть: 1

# Проверяем правила iptables
iptables -t nat -L -n -v

# На HQ-RTR — проверяем маршрут по умолчанию
sh ip route
# Должна быть строка: 0.0.0.0/0 via 172.16.1.1

# Пинг по шагам с HQ-SRV
ping 192.168.100.1   # → HQ-RTR (локальный шлюз)
ping 172.16.1.1      # → ISP
ping 8.8.8.8         # → интернет
❌ GRE-туннель не работает (нет пинга 10.10.10.2)
Причина: неверные local/remote адреса, брандмауэр блокирует GRE-пакеты (протокол 47), туннель не включён.
# На HQ-RTR — проверяем туннель
sh int tunnel gre 10
sh ip route

# Убедитесь, что enable выполнен для туннеля
configure
tunnel gre 10
enable
exit
commit
confirm

# Если брандмауэр блокирует — добавьте правило на ISP
iptables -I FORWARD -p gre -j ACCEPT
iptables-save > /etc/sysconfig/iptables
❌ OSPF не устанавливает соседство (нет FULL)
Причина: разные ключи аутентификации, туннель не работает, неверный router-id или network.
# Проверяем соседей
sh ip ospf neighbor
sh ip ospf

# Проверяем ключевую цепочку
sh key-chain

# Отлаживаем OSPF
debug ip ospf events
❌ SSH не принимает подключения на порт 2026
Причина: порт не открыт в firewalld, SELinux блокирует нестандартный порт, sshd не перезапущен.
# Проверяем, слушает ли sshd порт 2026
ss -tlnp | grep ssh

# Открываем порт в firewalld
firewall-cmd --permanent --add-port=2026/tcp
firewall-cmd --reload

# Если SELinux включён — разрешаем порт
semanage port -a -t ssh_port_t -p tcp 2026

# Перезапускаем sshd
systemctl restart sshd
systemctl status sshd
❌ BIND не запускается (named failed)
Причина: ошибки в синтаксисе named.conf или файлах зон, неверные права доступа.
# Проверяем конфигурацию
named-checkconf /etc/named.conf

# Проверяем каждую зону
named-checkzone au-team.irpo /var/named/master/au-team.irpo
named-checkzone 100.168.192.in-addr.arpa /var/named/master/100.168.192.in-addr.arpa

# Проверяем права
ls -la /var/named/master/

# Смотрим журнал ошибок
journalctl -u named -n 50
❌ EcoRouter не сохраняет изменения после перезагрузки
Причина: забыли выполнить commit и confirm.
# Всегда завершайте конфигурацию командами:
end
commit
confirm
# Без confirm изменения не сохраняются!
