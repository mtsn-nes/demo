Контроллер домена Samba DC
BR-SRV — домен sirius-exam.org (необязательное задание)
Задание Настроить Samba DC на BR-SRV: домен sirius-exam.org, ввести HQ-CLI в домен, создать 5 пользователей (hquser1–5), группу hq, настроить sudo для группы (только cat, grep, id).
⚠ Примечание Это задание объёмное и трудоёмкое. Если времени мало — пропустите и переходите к заданию 2. Задание 1 можно выполнить в последнюю очередь.

1.1 Подготовка HQ-SRV — разрешение зонной передачи

# На HQ-SRV — открываем named.conf
nano /etc/named.conf

Внутри блока options { ... } добавьте строку, разрешающую зонную передачу на BR-SRV:

allow-transfer { 192.168.255.2; };
# Перезапускаем DNS
systemctl restart named

1.2 Подготовка BR-SRV — SELinux и DNS

# Переводим SELinux в режим уведомлений (permissive)
nano /etc/selinux/config

Меняем SELINUX=enforcing на:

SELINUX=permissive
# Применяем без перезагрузки
setenforce 0

Настраиваем сетевой интерфейс BR-SRV — добавляем DNS-серверы:

# Прописываем HQ-SRV как первый DNS, BR-SRV как второй
nmcli con mod ens18 \
  ipv4.dns "192.168.100.2 192.168.255.2" \
  ipv4.dns-search "sirius-exam.org"
nmcli con up ens18

Отключаем конфликт systemd-resolved с Samba DNS:

nano /etc/systemd/resolved.conf

Найдите строку #DNSStubListener=yes и замените на:

DNSStubListener=no
# Применяем
systemctl restart systemd-resolved.service NetworkManager

# Проверяем — в выводе не должно быть 127.0.0.53
cat /etc/resolv.conf

1.3 Установка Samba (если не предустановлена)

📦 Пакеты samba samba-client samba-common уже предустановлены. Установка krb5* может потребоваться дополнительно.

1.4 Создание домена

# Сохраняем оригинальный smb.conf (он мешает samba-tool)
mv /etc/samba/smb.conf /etc/samba/smb.conf.bak

# Бэкап kerberos конфига
cp /etc/krb5.conf /etc/krb5.conf.bak

Настраиваем Kerberos — открываем /etc/krb5.conf:

nano /etc/krb5.conf

Приводим файл к следующему виду (полностью заменяем содержимое):

[logging]
 default = FILE:/var/log/krb5libs.log
 kdc = FILE:/var/log/krb5kdc.log
 admin_server = FILE:/var/log/kadmind.log

[libdefaults]
 dns_lookup_realm = false
 ticket_lifetime = 24h
 renew_lifetime = 7d
 forwardable = true
 rdns = false
 default_realm = SIRIUS-EXAM.ORG
 default_ccache_name = KEYRING:persistent:%{uid}

[realms]
SIRIUS-EXAM.ORG = {
 kdc = br-srv.sirius-exam.org
 admin_server = br-srv.sirius-exam.org
}

[domain_realm]
 .sirius-exam.org = SIRIUS-EXAM.ORG
 sirius-exam.org = SIRIUS-EXAM.ORG

Открываем и правим файл шифрования Kerberos:

nano /etc/krb5.conf.d/crypto-policies
[libdefaults]
default_tgs_enctypes = aes256-cts-hmac-sha1-96 aes128-cts-hmac-sha1-96 RC4-HMAC DES-CBC-CRC DES3-CBC-SHA1 DES-CBC-MD5
default_tkt_enctypes = aes256-cts-hmac-sha1-96 aes128-cts-hmac-sha1-96 RC4-HMAC DES-CBC-CRC DES3-CBC-SHA1 DES-CBC-MD5
preferred_enctypes = aes256-cts-hmac-sha1-96 aes128-cts-hmac-sha1-96 RC4-HMAC DES-CBC-CRC DES3-CBC-SHA1 DES-CBC-MD5

Запускаем полуавтоматическую настройку домена:

# Интерактивный режим — проверяем значения и нажимаем Enter
# Realm: SIRIUS-EXAM.ORG, Domain: SIRIUS-EXAM, DNS backend: SAMBA_INTERNAL
# DNS forwarder: 192.168.100.2
samba-tool domain provision --use-rfc2307 --interactive
# Запускаем Samba
systemctl enable --now samba

# Проверяем домен
samba-tool domain info 127.0.0.1

Файловое хранилище — RAID0
HQ-SRV — два диска по 1 Гб → /dev/md0 → /raid
Задание Создать RAID0 из двух дисков (1 Гб каждый), имя устройства md0, файловая система ext4, автомонтирование в /raid.
📦 Пакет mdadm уже предустановлен.

2.1 Определяем имена дисков

# Смотрим список всех дисков
lsblk

# Ожидаемый вывод: помимо sda (системный диск),
# должны быть sdb и sdc по 1G каждый
📌 Если диски называются иначе Если ваши дополнительные диски называются не sdb/sdc, а, например, vdb/vdc — подставьте их имена во все команды ниже.

2.2 Создание RAID0

# Создаём массив RAID0 из двух дисков
# -l 0 — уровень RAID0 (страйп), -n 2 — количество дисков
mdadm --create --verbose /dev/md0 -l 0 -n 2 /dev/sdb /dev/sdc

# Проверяем — у sdb и sdc должен появиться md0
lsblk

2.3 Сохранение конфигурации массива

# Сохраняем конфигурацию в mdadm.conf (чтобы массив восстанавливался после перезагрузки)
mdadm --detail --scan --verbose >> /etc/mdadm.conf

# Проверяем содержимое
cat /etc/mdadm.conf

2.4 Форматирование и монтирование

# Форматируем RAID в ext4
mkfs.ext4 /dev/md0

# Создаём директорию для монтирования
mkdir /raid

# Монтируем вручную (для проверки)
mount /dev/md0 /raid

# Проверяем — /dev/md0 должен быть примонтирован в /raid
lsblk -f
df -h

2.5 Автомонтирование через fstab

# Открываем fstab
nano /etc/fstab

Добавляем в конец файла строку:

/dev/md0    /raid    ext4    defaults    0    0
# Проверяем fstab без перезагрузки
mount -av
# Строка для /raid должна показать "already mounted" или "successfully mounted"
✅ Ожидаемый результат df -h → строка /dev/md0 2.0G ... /raid
После перезагрузки: lsblk → md0 примонтирован в /raid автоматически.
3
NFS-сервер
HQ-SRV экспортирует /raid/nfs → HQ-CLI монтирует в /mnt/nfs
Задание Папка общего доступа /raid/nfs на HQ-SRV, доступ rw только для сети 192.168.200.0/28 (HQ-CLI). На HQ-CLI — автомонтирование в /mnt/nfs.
📦 Пакеты nfs4-acl-tools и nfs-utils уже предустановлены.

3.1 Настройка сервера NFS (HQ-SRV)

# Создаём общую папку внутри RAID-массива
mkdir /raid/nfs

# Открываем права (чтение и запись для всех)
chmod -R 777 /raid/nfs

Создаём экспорт NFS — открываем файл /etc/exports:

nano /etc/exports

Добавляем строку:

/raid/nfs    192.168.200.0/28(rw,no_root_squash)

Параметры: rw — чтение и запись, no_root_squash — root на клиенте имеет права root на шаре.

# Запускаем NFS-сервер и добавляем в автозагрузку
systemctl enable --now nfs-server.service

# Применяем экспорты (читаем /etc/exports)
exportfs -arv
# Ожидаем: exporting 192.168.200.0/28:/raid/nfs

# Проверяем активные экспорты
exportfs -v

3.2 Настройка клиента NFS (HQ-CLI)

# На HQ-CLI — создаём точку монтирования
mkdir /mnt/nfs

chmod -R 777 /mnt/nfs

Добавляем автомонтирование в fstab на HQ-CLI:

nano /etc/fstab

Добавляем строку:

192.168.100.2:/raid/nfs    /mnt/nfs    nfs    auto    0    0
# Монтируем всё из fstab
mount -av
# Должна появиться строка: /mnt/nfs mounted successfully

# Проверяем
df -h | grep nfs

3.3 Проверка работы

# На HQ-SRV — создаём тестовый файл
touch /raid/nfs/test.txt

# На HQ-CLI — проверяем что файл виден
ls /mnt/nfs/
# Должны увидеть: test.txt
✅ Ожидаемый результат Файлы, созданные на HQ-SRV в /raid/nfs, мгновенно видны на HQ-CLI в /mnt/nfs, и наоборот.
4
Служба сетевого времени (NTP)
Сервер chrony на ISP, клиенты: HQ-SRV, HQ-CLI, BR-RTR, BR-SRV
Задание NTP-сервер на ISP (stratum 5), вышестоящий сервер ntp1.vniiftri.ru. Клиенты: HQ-SRV, HQ-CLI, BR-RTR, BR-SRV (синхронизируются с ISP).

4.1 Настройка NTP-сервера на ISP

# Открываем конфиг chrony на ISP
nano /etc/chrony.conf

Приводим файл к следующему виду (оставляем один сервер, остальные комментируем, добавляем stratum и allow):

# Оставляем один вышестоящий сервер, добавляем iburst prefer minstratum 4
server ntp1.vniiftri.ru iburst prefer minstratum 4

# Остальные серверы ЗАКОММЕНТИРОВАТЬ (поставить # перед строками)
#server ntp2.vniiftri.ru iburst
#server ntp3.vniiftri.ru iburst
#server ntp4.vniiftri.ru iburst

# Добавляем в конец файла:
local stratum 5
allow 0.0.0.0/0

local stratum 5 — ISP будет выдавать время с уровнем 5, даже если не синхронизирован с вышестоящим. allow 0.0.0.0/0 — разрешаем синхронизацию всем клиентам.

# Перезапускаем chrony
systemctl restart chronyd

# Проверяем синхронизацию с вышестоящим сервером
chronyc sources
# Ожидаем: строка с ntp1.vniiftri.ru, stratum 4–5

4.2 Настройка клиентов офиса HQ (HQ-SRV и HQ-CLI)

Выполнить на обоих устройствах:

# Открываем конфиг chrony
nano /etc/chrony.conf

Комментируем все строки server ... по умолчанию и добавляем ISP:

# Комментируем все существующие строки server/pool
#server ntp1.vniiftri.ru iburst
#server ntp2.vniiftri.ru iburst
#server ntp3.vniiftri.ru iburst
#server ntp4.vniiftri.ru iburst

# Добавляем ISP как NTP-сервер
server 172.16.1.1 iburst prefer
# Перезапускаем
systemctl restart chronyd

# Проверяем — должен показать ISP (172.16.1.1) со stratum 5
chronyc sources

4.3 Настройка клиентов офиса BR (BR-RTR и BR-SRV)

⚠ BR-RTR тоже клиент NTP! По заданию клиенты NTP: HQ-SRV, HQ-CLI, BR-RTR, BR-SRV. BR-RTR настраивается через Eltex CLI.

BR-RTR (Eltex ESR)

# На BR-RTR через консоль Proxmox или SSH
configure

ntp server 172.16.2.1
  enable
exit

end
commit
confirm

# Проверка
show ntp status

BR-SRV (Linux)

nano /etc/chrony.conf

Аналогично HQ, но ISP-интерфейс для BR:

#server ntp1.vniiftri.ru iburst
#server ntp2.vniiftri.ru iburst
#server ntp3.vniiftri.ru iburst
#server ntp4.vniiftri.ru iburst

server 172.16.2.1 iburst prefer
systemctl restart chronyd
chronyc sources
✅ Ожидаемый результат На HQ-SRV/HQ-CLI: chronyc sources показывает * 172.16.1.1 со stratum 5.
На BR-SRV: chronyc sources показывает * 172.16.2.1 со stratum 5.
На BR-RTR: show ntp status → синхронизирован с 172.16.2.1.
5
Ansible
BR-SRV — инвентарь, ping всех хостов
Задание Ansible на BR-SRV, рабочий каталог /etc/ansible, инвентарь: HQ-SRV, HQ-CLI, HQ-RTR, BR-RTR. Все хосты отвечают pong на ping без ошибок.

5.1 Установка Ansible

# Устанавливаем Ansible и sshpass (нужен для аутентификации по паролю)
dnf install ansible sshpass -y

5.2 Создание файла инвентаря

# Открываем файл инвентаря
nano /etc/ansible/hosts

Содержимое файла (используем ваши IP-адреса):

[hq]
# HQ-RTR (Eltex ESR) — подключаемся как net_admin на стандартный SSH
192.168.100.1 ansible_user=net_admin ansible_password=P@ssw0rd

# HQ-SRV — подключаемся как sshuser через порт 2026
192.168.100.2 ansible_user=sshuser ansible_password=P@ssw0rd ansible_port=2026

# HQ-CLI — подключаемся как user
192.168.200.2 ansible_user=user ansible_password=user

[br]
# BR-RTR (Eltex ESR) — подключаемся как net_admin
192.168.255.1 ansible_user=net_admin ansible_password=P@ssw0rd
📌 IP HQ-CLI Если HQ-CLI получает адрес по DHCP, уточните текущий IP командой ip a на HQ-CLI и замените 192.168.200.2 на фактический.

5.3 Настройка ansible.cfg

nano /etc/ansible/ansible.cfg

Добавляем или находим секцию [defaults] и прописываем:

[defaults]
host_key_checking = False
interpreter_python = auto_silent

host_key_checking = False — отключает проверку SSH-ключей хостов (нет запроса "добавить в known_hosts"). interpreter_python = auto_silent — убирает предупреждения об обнаружении Python.

5.4 Проверка связности

# Запускаем ping на всех хостах
ansible all -m ping

# Ожидаемый результат для каждого хоста:
# 192.168.100.2 | SUCCESS => {
#     "ping": "pong"
# }
⚠ Если HQ-RTR или BR-RTR не отвечает на ping Eltex ESR может не иметь Python, необходимого для Ansible. В этом случае для маршрутизаторов используйте модуль raw вместо ping:
ansible 192.168.100.1 -m raw -a "hostname"
Если RTR вообще не доступен через SSH от BR-SRV — убедитесь что маршрут до 192.168.100.1 существует (через GRE-туннель).

BR-SRV — контейнеры testapp (порт 8080) + db (MariaDB)
Задание Docker на BR-SRV: контейнер testapp (site:latest, порт 8080) + db (mariadb:10.11, порт 3306). БД: testdb, пользователь test / P@ssw0rd. Приложение доступно снаружи через 192.168.255.2:8080.
⚠ В задании написано: container_name должно быть tespapp (не testapp!) — это опечатка в задании, но на экзамене называй именно так!
📦 Пакеты docker-ce, docker-ce-cli, docker-compose уже предустановлены.

6.1 Запуск Docker

# Запускаем Docker и добавляем в автозагрузку
systemctl enable docker --now

# Проверяем статус
systemctl status docker

6.2 Загрузка образов

Способ А — загрузка из tar-архивов в /root (если образы уже скачаны):

# Смотрим какие архивы есть
ls -l /root/

# Импортируем образы (имена файлов могут отличаться)

# Либо с Additional.iso (если диск подключён)

docker load < /mnt/docker/mariadb_latest.tar

Способ Б — скачать через интернет (если есть доступ):

docker pull mariadb:10.11
# site:latest — специфический образ из задания, может быть только на iso
# Проверяем список образов
docker image ls
# Должны быть: site:latest и mariadb:10.11

6.3 Создание docker-compose файла

nano /root/web.yaml

Полное содержимое файла (используем IP BR-SRV = 192.168.255.2):

services:
  testapp:
    container_name: tespapp
    image: site:latest
    restart: always
    ports:
      - "8080:8000"
    environment:
      DB_HOST: "192.168.255.2"
      DB_PORT: "3306"
      DB_NAME: testdb
      DB_USER: test
      DB_PASS: P@ssw0rd
      DB_TYPE: maria
    depends_on:
      - db

  db:
    container_name: db
    image: mariadb:10.11
    restart: always
    ports:
      - "3306:3306"
    environment:
      DB_USER: test

Статическая трансляция портов (DNAT)
HQ-RTR и BR-RTR — проброс портов 8080 и 2026
Задание HQ-RTR: 8080→HQ-SRV:80, 2026→HQ-SRV:2026.
BR-RTR: 8080→BR-SRV:8080, 2026→BR-SRV:2026.
⚠ Eltex ESR — синтаксис DNAT В вашей топологии HQ-RTR и BR-RTR — это Eltex ESR, а не Linux. Поэтому используется синтаксис Eltex ESR, а не nftables.

8.1 Проброс портов на HQ-RTR (Eltex ESR)

Подключаемся к HQ-RTR и настраиваем DNAT:

# Входим в режим конфигурации
configure

# Создаём пул адресов для DNAT на HQ-SRV
nat destination
pool HQ_SRV_WEB
ip address-range 192.168.100.2
exit

# Создаём набор правил DNAT
ruleset DNAT_HQ
from zone UNTRUST

# Правило 1: 8080 снаружи → 80 на HQ-SRV (веб-приложение Apache)
rule 1
match destination-port 8080
match protocol tcp
action destination-nat pool HQ_SRV_WEB port 80
enable
exit

# Правило 2: 2026 снаружи → 2026 на HQ-SRV (SSH)
rule 2
match destination-port 2026
match protocol tcp
action destination-nat pool HQ_SRV_WEB port 2026
enable
exit
exit
exit

end
commit
confirm

Возможные ошибки и их решение
Диагностика типовых проблем Модуля 2
❌ RAID не создаётся: mdadm: error — not enough devices
Причина: диски уже разбиты на разделы или содержат данные.
# Очищаем диски перед созданием RAID
wipefs -a /dev/sdb
wipefs -a /dev/sdc
# Повторяем команду mdadm --create
❌ NFS: mount.nfs: Connection timed out на HQ-CLI
Причина: firewall на HQ-SRV блокирует NFS, или nfs-server не запущен.
# На HQ-SRV — проверяем статус
systemctl status nfs-server

# Открываем NFS в firewall
firewall-cmd --permanent --add-service=nfs
firewall-cmd --permanent --add-service=rpcbind
firewall-cmd --permanent --add-service=mountd
firewall-cmd --reload

# Проверяем экспорты
exportfs -v
❌ Docker: контейнер testapp постоянно перезапускается
Причина: база данных testdb не создана или пользователь test не имеет прав.
# Смотрим логи контейнера
docker logs tespapp

# Заходим в db и проверяем базы
docker exec -it db mysql -u root -p
SHOW DATABASES;
SHOW GRANTS FOR 'test'@'%';
\q

# После создания базы — перезапускаем стек
docker-compose -f /root/web.yaml restart
❌ Apache: 403 Forbidden или пустая страница
Причина: SELinux мешает httpd читать файлы, или неправильные права на index.php.
# Устанавливаем права на файлы Apache
chmod 644 /var/www/html/index.php
chown apache:apache /var/www/html/index.php

# SELinux — переводим в permissive если не сделали
setenforce 0

# Проверяем логи Apache
tail -50 /var/log/httpd/error_log
❌ nginx -t: unknown directive "server" (при добавлении виртуальных хостов)
Причина: новые блоки server добавлены за пределами блока http { }, или неправильно расставлены скобки.
# Проверяем структуру nginx.conf — блоки server должны быть ВНУТРИ http { }
# Правильная структура:
# http {
#   server { ... }   ← существующий
#   server { ... }   ← наш для web
#   server { ... }   ← наш для docker
# }

# Смотрим конец файла — наши блоки должны быть ДО последней}
tail -30 /etc/nginx/nginx.conf
❌ htpasswd: command not found
Причина: пакет httpd-tools не установлен (хотя должен быть предустановлен).
dnf install httpd-tools -y
❌ Ansible: UNREACHABLE — Connection timed out
Причина: хост недоступен по сети, или SSH-порт закрыт, или неверные учётные данные.
# Проверяем доступность хоста
ping -c 3 192.168.100.2

# Запускаем ping с подробным выводом
ansible 192.168.100.2 -m ping -vvv

Причина: нет связи с вышестоящим NTP-сервером, или неверный IP сервера в chrony.conf.
# На ISP — проверяем что сервер слышит вышестоящий
